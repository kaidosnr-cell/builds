  ─────────────────────────────────────────────────────────────────────────────
  LAYER 1 — STATIC PROTECTIONS  (binary at rest)
  ─────────────────────────────────────────────────────────────────────────────

  [1.1]  STRING OBFUSCATION
         Compile-time encryption of sensitive string literals via C++ templates
         (XOR or AES-128 with a key generated at compile time). Strings are
         decrypted only at the moment of use — never stored in plaintext in
         .rdata or .data sections.

         Requirements:
           · List every sensitive string found and which must be encrypted.
           · Provide a header-only implementation compatible with the detected
             C++ standard.
           · Suggested pattern:
               template<size_t N, uint8_t Key>
               struct ObfStr { constexpr ObfStr(...); const char* decrypt(); };
           · The decryption key must NOT be trivially visible — derive it from
             a compile-time expression, not a raw constant.
         Performance impact: LOW

  [1.2]  IMPORT TABLE OBFUSCATION (IAT Hardening)
         Dynamic resolution of critical Windows APIs via GetProcAddress and
         function name hashing instead of direct imports.

         Requirements:
           · Identify the most revealing imports in this project's IAT
             (e.g., CryptEncrypt, RegOpenKeyEx, WSAConnect).
           · Use FNV-1a or djb2 hashing of API names at compile time.
           · Implement a custom GetProcAddressByHash helper.
         Performance impact: LOW to MODERATE

  [1.3]  COMPILER FLAGS AND BINARY STRIPPING
         Configure the release build to minimize information leakage.

         Requirements — apply ALL of the following:
           · Strip all debug symbols
               MSVC:  /DEBUG:NONE
               GCC:   -s
           · Disable RTTI selectively on critical modules (-fno-rtti)
           · Enable LTO/LTCG for less structured, harder-to-follow code
           · Remove .pdb references and all source path information
               GCC/Clang: -fmacro-prefix-map=OLD=NEW
               MSVC:      /d1trimfile
           · Enable dead code elimination and COMDAT folding
               MSVC:  /OPT:REF  /OPT:ICF
           · Propose EXACT modifications to CMakeLists.txt / .vcxproj / Makefile
         Performance impact: NONE to NEGATIVE (improves release performance)

  [1.4]  ANTI-DISASSEMBLY  (apply selectively to [CRITICAL] and [HIGH] only)
         Insert ambiguous opcode sequences that mislead linear disassemblers
         (IDA, Ghidra) without affecting execution:
           · Confusion bytes between valid instructions
           · Unconditional jumps to the middle of a multi-byte instruction
           · Static opaque predicates to break the control flow graph
           · Junk instruction insertion between meaningful blocks

         Cite explicitly which functions from the risk assessment receive this.
         Do not apply globally — increased binary size without proportional
         benefit outside critical functions.
         Performance impact: LOW

  [1.5]  SECTION ENCRYPTION / PACKING  (evaluate per project)
         Evaluate whether a lightweight custom packer or modified UPX
         (with custom stubs defeating known signatures) fits the deployment.

         If deployment constraints exist (signed installers, AV whitelisting,
         code signing certificates, auto-update), document why a packer would
         or would not be appropriate.

         If applicable, propose a custom stub loader that decrypts the .text
         section at startup using a key derived from a hardware fingerprint.
         Performance impact: MODERATE (startup overhead)

  ─────────────────────────────────────────────────────────────────────────────
  LAYER 2 — DYNAMIC PROTECTIONS  (runtime analysis resistance)
  ─────────────────────────────────────────────────────────────────────────────

  [2.1]  MULTI-LAYER DEBUGGER DETECTION
         Implement a centralized AntiDebug class. Select checks based on the
         detected target platform.

         ── WINDOWS ──────────────────────────────────────────────────────────
         · IsDebuggerPresent()
         · CheckRemoteDebuggerPresent()
         · NtQueryInformationProcess(ProcessDebugPort)     [dynamic resolution]
         · NtQueryInformationProcess(ProcessDebugFlags)    [dynamic resolution]
         · PEB.NtGlobalFlag check:
             x86: fs:[0x30]+0x68   |   x64: gs:[0x60]+0xBC
             Value 0x70 → debugger present
         · PEB heap flags: ForceFlags != 0 under debugger
         · Hardware breakpoint detection via GetThreadContext():
             inspect DR0–DR3, DR6, DR7 — any non-zero DR0-DR3 = breakpoint set
         · Self-INT3 scan: scan critical function prologues for 0xCC bytes
         · Timing attack:
             delta between two RDTSC / QueryPerformanceCounter calls;
             threshold > 500ms (configurable) indicates single-stepping
         · Debugger window detection via FindWindowA/W against known titles:
             x64dbg, OllyDbg, WinDbg, Cheat Engine, IDA, Ghidra,
             Binary Ninja, x32dbg, dnSpy, ReClass.NET
         · SeDebugPrivilege check in current process token
         · Heap handle check: HeapWalk on default heap differs under debugger
         · CloseHandle(invalid_handle) via SEH:
             if exception caught → debugger present
         · OutputDebugString timing trick (layered value)

         ── LINUX ─────────────────────────────────────────────────────────────
         · /proc/self/status → parse TracerPid; non-zero = traced
         · ptrace(PTRACE_TRACEME, 0, 0, 0) returns -1 if already traced
         · /proc/self/wchan → "ptrace_stop" string indicates tracing
         · Timing delta via clock_gettime(CLOCK_MONOTONIC)
         · Check parent process name (/proc/ppid/comm) against known debuggers

         ── macOS ─────────────────────────────────────────────────────────────
         · sysctl(KERN_PROC_PID) → kinfo_proc.p_flag & P_TRACED
         · ptrace(PT_DENY_ATTACH, 0, 0, 0) at process start
         · Timing delta via mach_absolute_time()

         ── IMPLEMENTATION REQUIREMENTS ───────────────────────────────────────
         · For EACH check: implement a discreet, non-trivial response (see 2.3).
           Do NOT call ExitProcess() / exit() directly — too easily patchable.
         · Wrap all sensitive API calls in dynamic resolution.
           No direct IAT entries for NtQueryInformationProcess, etc.
         · Randomize check execution order at runtime to resist automated bypass.
         · Run checks from multiple threads to prevent single-point patching.
         Performance impact: LOW

  [2.2]  MEMORY ANALYSIS AND HOOKING DETECTION

         A. IAT INTEGRITY CHECK
            Verify the first N bytes of critical imported functions match
            expected opcodes. Detect JMP patches / inline hooks.

         B. HYPERVISOR / SANDBOX DETECTION
            · CPUID leaf 0x1, bit 31 (hypervisor present bit)
            · Abnormal timing on privileged instructions
            · Absence of expected physical devices (registry / WMI queries)
            · Low physical memory / low CPU count heuristics

         C. INJECTED DLL DETECTION
            · EnumProcessModules / /proc/self/maps scan against a whitelist
            · Flag any non-whitelisted module as suspicious
            · Specifically detect known tool signatures:
                frida-agent, minhook, detours, scyllahide

         D. SELF .TEXT INTEGRITY SCAN
            · On startup: compute checksum (xxHash or CRC32) of each critical
              module's .text section
            · Periodically re-verify on a background thread with randomized
              intervals
            · On mismatch → ThreatResponse::COMPROMISED

         Run all detection asynchronously. Randomize polling intervals
         (e.g., 15–90 seconds) to avoid detectable timing patterns.
         Performance impact: MODERATE

  [2.3]  DEFERRED AND STEALTHY THREAT RESPONSE STRATEGY
         Core principle: An immediate crash is trivially identified and patched.
         Use graduated degradation instead.

         Implement a thread-safe singleton ThreatResponse with:
           enum class ThreatLevel { CLEAN, SUSPICIOUS, COMPROMISED };

         ON SUSPICIOUS:
           · Start a hidden timer (random delay: 2–10 minutes)
           · Log detection details to an encrypted internal buffer
           · Begin subtly corrupting non-obvious internal state

         ON COMPROMISED:
           · Silently corrupt a critical internal state variable after a
             random delay (30 seconds to 5 minutes post-detection)
           · Produce subtly incorrect results in key algorithms
             (wrong output values, off-by-one in critical calculations)
           · Trigger abnormal but non-obvious application behavior
           · Optionally write an encrypted tamper log to disk or registry

         Rules:
           · NEVER display any error message referencing protection or tampering.
           · Corruption must make the analyst believe the program is buggy,
             not protected — wasting their time significantly.
           · The ThreatResponse module must itself be integrity-checked.
         Performance impact: NEGLIGIBLE

  [2.4]  IN-MEMORY CODE INTEGRITY VERIFICATION
         · At initialization AND periodically (dedicated thread, randomized
           interval): compute xxHash128 or CRC32C of .text sections of
           critical in-memory modules; compare against reference values.
         · Reference hashes embedded at build time via a post-build script
           that computes them from compiled object files.
         · On divergence → ThreatResponse::COMPROMISED.
         · The integrity check thread must verify its own code as well.
         Performance impact: LOW  (xxHash128 ≈ 50 GB/s)

  [2.5]  ANTI-MEMORY DUMP
         · Use VirtualLock on pages containing the most critical algorithms
           to prevent paging and complicate memory acquisition.
         · Implement a decrypt-on-use pattern for infrequently called
           [CRITICAL] functions:
             - Keep the function encrypted in memory
             - Decrypt to an executable page immediately before execution
             - Execute
             - Re-encrypt and mark the page PAGE_NOACCESS
             (use VirtualProtect PAGE_EXECUTE_READ / PAGE_NOACCESS cycling)
         · For performance-critical [CRITICAL] functions: use inline data
           decryption combined with Layer 1 anti-disassembly instead.
         Performance impact: MODERATE to HIGH depending on call frequency

  [3.2]  CONTROL FLOW FLATTENING (CFF)
         Transform the CFG of [CRITICAL] and [HIGH] functions into an
         obfuscated switch-case dispatch driven by a state variable, making
         the CFG useless for static analysis.

         Propose either:
           a) A manual macro/template C++ implementation for the identified
              functions — implement it FULLY, not as pseudocode, OR
           b) Integration of OLLVM / Hikari as a Clang pass if the project
              uses Clang — provide the exact CMake integration commands.

         The dispatching variable must be obfuscated with MBA (see 3.3).
         Performance impact: MODERATE

  [3.3]  MIXED BOOLEAN ARITHMETIC (MBA) OBFUSCATION
         Replace arithmetic and logical expressions in sensitive algorithms
         with semantically equivalent MBA expressions that resist symbolic
         simplifiers (Miasm, angr, Z3, Triton).

         Examples:
           a ^ b       →  (a | b) - (a & b)
           a + b       →  (a ^ b) + 2*(a & b)
           a * 2       →  (a << 1) | (a >> 31) & 0   [with noise]
           constant K  →  ((K ^ mask) + offset) - offset ^ mask

         Apply ONLY to [CRITICAL] functions. Implement as inline helper
         functions or macros. Focus on: keys, state variables, magic constants,
         checksums — the values most informative to an analyst.
         Performance impact: LOW to MODERATE

  [3.4]  BUILD WATERMARKING
         Embed a unique, invisible fingerprint in every release build:
           · Functionally neutral instruction sequences in non-critical code
             that statistically identify the build origin.
           · Magic constants encoding a build ID folded into calculations
             (e.g., PRNG seed values, initial hash states).
           · A unique per-build constant in dead branches that never execute
             but survive stripping.

         Provide a concrete CMake script that:
           · Generates a unique build UUID at configuration time
           · Embeds it via a generated header
           · Uses it in at least 3 independent binary locations

         Document the forensic recovery procedure for identifying a
         watermarked build from a stripped binary.
